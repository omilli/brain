# Memory

The persistent self-improvement layer, stored as a markdown knowledge bundle. `memory/entries/*.md` are concept documents (the canonical source); `memory/archive/` holds retired ones; `index.md` is a **derived** directory-listing regenerated by `python3 .agents/skills/brain-memory/memory.py rebuild` (never hand-edited); `log.md` is the update history. The `brain-prime` skill loads concepts on trigger. This skill owns the write side — it never loads on every run, only when a memory event needs capturing. Conservative like `brain-feedback`: most events fail the write gate; that is the correct, common outcome.

The knowledge base lives at the repo root (`<repo>/memory/`), separate from this skill folder (`.agents/skills/brain-memory/`) so a pack update never clobbers live data.

**Concept schema** — frontmatter keys:

    ---
    type: decision                                  # REQUIRED. correction for a fix
    title: Imperative title (the rule/fact itself)
    description: One-sentence actionable core.      # loads by default; rebuild inlines it into index.md
    tags: [arch, contract]                          # coarse scoping
    timestamp: 2026-05-02                           # last meaningful change
    last_confirmed: 2026-05-02                      # bump on every re-verification (staleness surface)
    triggers: [api-contract, rename-symbol]         # 2–4 task words; the discovery surface (avoid bare common words that over-fire)
    supersedes: 009                                 # ONLY if retiring another concept
    ---
    # Why       rationale / alternatives / what breaks if ignored
    # Evidence  source citation, the passing command, or the error it fixes

Concept ID = filename stem. `title`/`description` live in frontmatter, not the body. Quote any frontmatter value containing `: `, leading YAML indicators, or `"` — `memory.py` parses both bare and double-quoted scalars.

**Command form.** The script travels with this skill at `.agents/skills/brain-memory/memory.py` — it is vendored, not kept external. Invoke it as `python3 .agents/skills/brain-memory/memory.py <cmd> [--root <dir>]`. It resolves the repo knowledge base automatically (`<git-toplevel>/memory`), so a project-scoped write needs no `--root`. Pass `--root <dir>` only to target an alternate store (for example a personal global KB you maintain outside this pack). This mirrors how every other skill gets into a repo: copied in, one canonical copy per repo, no drift across projects.

## Step 1 — Confirm it is a memory event, not a feedback event

A memory event fires when, during a run, one of these produced a durable, recallable learning:
- A retry after a failure revealed the correct approach.
- A non-obvious fact was confirmed against source/docs (not carried from memory).
- A deviation from a stated rule was necessary and justified.
- The user corrected you.

Route check — `brain-feedback` owns it if the learning should change always-on text in `AGENTS.md` or a skill (a rule change). `brain-memory` owns it if it is a decision or fact worth recalling on demand but not a rule. One sink per event; if both apply, feedback takes the rule, memory takes the residual fact.

## Step 2 — Apply the write gate

Write only if ALL hold:
- Verified this session against source, a passing check, or an explicit user statement — not inference or memory.
- Load-bearing: recalling it later would change an action or avoid rework.
- Not already captured (grep `memory/` first; duplicates merge rather than multiply).

If any fails, do not write.

## Step 3 — Write the concept

File: `memory/entries/NNN.md` — run `python3 .agents/skills/brain-memory/memory.py add` to allocate the next id and write the template (or `--fix` for a `type: correction`). The fixed frontmatter+body order is the template above. Triggers/tags/description live in the concept file itself (single source) — NOT in index.md. After writing or editing any concept, always regenerate the derived index: `python3 .agents/skills/brain-memory/memory.py rebuild`. The write commands (`add`, `supersede`, `log`) also append to `log.md`. Forgetting rebuild is the only remaining failure mode, so it closes every write.

## Step 4 — Supersede on contradiction (one-in-one-out)

If the new concept contradicts or retires an active one, run one command:

    python3 .agents/skills/brain-memory/memory.py supersede <old> <new>

It moves `entries/<old>.md` → `archive/`, sets `supersedes: <old>` on the new concept's frontmatter, appends a Deprecation line to `log.md`, and rebuilds index.md. Location is the retire signal; no status field. Never leave two active concepts that disagree.

## Step 5 — Refresh on staleness

Run `python3 .agents/skills/brain-memory/memory.py stale` to list concepts with `last_confirmed` older than 180 days (default; pass a day count to override). For each: re-verify against current source before trusting it. Still true → bump `last_confirmed` and `timestamp`, then `python3 .agents/skills/brain-memory/memory.py rebuild`. False → run Step 4 (supersede) and act on the new truth. Loading is the trigger; no scheduled maintenance.

## Step 6 — Prune archive orphans

Archive exists only to serve the live supersession narrative (an active concept's `supersedes:` → one read hop). Concepts referenced by no active entry are dead weight git already holds, and greppable stale-truth is a footgun. After a supersession chain settles:

    python3 .agents/skills/brain-memory/memory.py prune            # dry-run: list orphans
    python3 .agents/skills/brain-memory/memory.py prune --apply    # delete them

`--apply` is a deletion — run dry-run first and apply only with intent.

## Self-check

a. Did I confirm this is a memory event, not a rule change for feedback?
b. Did the write gate pass (verified + load-bearing + non-duplicate)?
c. Does frontmatter carry only earning fields (`type`/`title`/`description`/`tags`/`timestamp` + extensions), body follows `# Why` → `# Evidence`?
d. If it retires a concept: did `memory.py supersede` run (archive move + `supersedes:` + `log.md` + rebuild) — no two active disagreements?
e. If loaded stale: re-verified and `last_confirmed` bumped or superseded?
f. Did the write close with `memory.py rebuild`? (the one remaining failure mode)
g. Did I leave orphaned archive entries? (`memory.py prune` after supersession chains)
